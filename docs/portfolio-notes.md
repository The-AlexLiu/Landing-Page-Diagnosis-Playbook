# 落地页内容诊断：从投放承接到用户任务的经验沉淀

生成日期：2026-06-24  
用途：作品集方法论页 / 项目复盘页 / 面试作品说明  
适用页面：广告落地页、商品页、活动页、SaaS 官网、App 下载页、B2B 线索页、内容页、服务办理页、平台产品介绍页

---

## 1. 我为什么要沉淀这套方法

过去看落地页时，很容易把问题说成“页面不够好看”“卖点不够突出”“按钮不够明显”。这些说法通常没错，但不够可执行，也很难说服投放、产品、设计、运营或老板一起改。

我后来更倾向于把落地页当成一个“承接用户任务的页面”来看。

用户不是无缘无故来到页面的。他可能刚点了 Meta 素材，刚搜了一个 Google 关键词，刚从邮件里点进来，刚在站内看到一个 Banner，或者刚被销售发了一个链接。他点进来时已经带着预期：

- 这个东西是不是我刚才看到的那个？
- 它到底能解决我什么问题？
- 价格、风险、售后、隐私这些事情清不清楚？
- 我下一步应该买、注册、下载、咨询，还是先再看看？
- 这家公司/产品/页面值不值得信任？

所以我做落地页诊断时，不会先问“这个页面漂不漂亮”，而是先问：

> 这个页面有没有接住用户点进来时的那个问题？

这套方法的价值在于，它不只适合独立站商品页，也适合更广泛的互联网产品页面。比如一个 SaaS 官网、一个 App 下载页、一个产品功能发布页、一个线索收集页、一个服务办理页，本质上都需要完成同一件事：让用户快速理解、相信，并顺利走到下一步。

---

## 2. 我的核心判断：落地页不是单页销售文案，而是转化链路的一段

如果只看页面本身，很容易陷入主观讨论：

- 你觉得标题不够强；
- 我觉得图片不够高级；
- 设计觉得视觉层级没问题；
- 投放觉得素材已经吸引点击了；
- 产品觉得功能都写上了。

但用户不会这样分工理解。用户看到的是一条连续链路：

> 入口承诺 → 页面首屏 → 内容解释 → 信任证据 → 行动路径 → 提交/购买/注册后的反馈

只要其中任何一段断了，转化就会掉。

举几个常见情况：

| 表现 | 表面看起来 | 我会优先怀疑 |
|---|---|---|
| CTR 高但 CVR 低 | 素材很吸引人，页面不转化 | 广告承诺和首屏内容断裂 |
| 页面滚动深但按钮点击少 | 用户有兴趣但不行动 | CTA 不匹配用户阶段，或风险没有解除 |
| 加购高但购买低 | 商品有吸引力但订单少 | 运费、退换、价格、支付信任有问题 |
| 表单页访问多但提交少 | 线索质量差 | 表单过重、隐私说明弱、提交后收益不清楚 |
| Google Ads 落地页体验差 | 质量分问题 | 关键词、广告、页面内容和导航没有对齐 |
| 用户问客服的问题很重复 | 页面内容不完整 | FAQ、价格、规则、售后信息没有前置 |

这也是为什么我会把落地页诊断拆成“内容、信任、路径、数据”几个部分，而不是只改标题或按钮。

## 3. 我会先看 4 个问题

正式评分前，我会先做一个快速判断。这个判断不用一开始就陷进所有细节，而是先定位页面和入口之间有没有明显断点。

### 3.1 用户从哪里来

不同入口带来的用户预期完全不同：

| 来源 | 用户预期 | 页面最应该接住什么 |
|---|---|---|
| Google Search | 我主动搜索了一个问题或商品 | 关键词意图、具体答案、可比较信息 |
| Google Shopping / PMax | 我看到了商品图、价格、标题 | 同款商品、价格一致、库存和优惠 |
| Meta Ads | 我被素材角度吸引 | 素材里的痛点、场景、达人/UGC、优惠 |
| TikTok / Reels | 我被短视频情绪和画面打动 | 同款画面、快速利益点、移动端体验 |
| EDM / 私域 | 我已经对品牌有一点认知 | 活动规则、会员权益、专属优惠 |
| SEO 内容 | 我想了解或比较 | 结构化答案、可信来源、下一步推荐 |
| 销售发来的链接 | 我在评估方案 | 案例、价格、资质、交付边界 |
| 站内 Banner | 我已经在品牌站内 | 明确活动、商品集合或任务入口 |

如果页面没有接住来源预期，后面的优化都会很吃力。

### 3.2 首屏能不能在几秒内讲清楚

我会看首屏有没有回答 5 个问题：

| 问题 | 页面应该回答 |
|---|---|
| 这是什么？ | 产品、服务、活动或任务入口 |
| 给谁用？ | 人群、角色、场景 |
| 解决什么问题？ | 用户利益，不只是品牌口号 |
| 为什么值得继续？ | 差异点、证据、结果、优惠 |
| 下一步做什么？ | 主 CTA，必要时给一个低承诺 CTA |

很多页面首屏看起来很完整，但实际只是在表达品牌态度。用户看完还是不知道“这和我有什么关系”。这种页面往往不适合直接承接冷流量。

### 3.3 用户为什么会不信

落地页不是把卖点写满就行。用户心里会不断冒出反对意见：

- 这个评价是真的吗？
- 价格为什么是这个水平？
- 有没有隐藏费用？
- 不合适能退吗？
- 数据安全吗？
- 订阅能取消吗？
- 这个案例和我有什么关系？
- 我提交表单后会被频繁骚扰吗？

页面如果不主动回答这些问题，用户就会用自己的方式降低风险：退出、比价、问客服、拖延、去搜品牌词，或者直接不买。

### 3.4 下一步是否顺手

很多页面的问题不是没有 CTA，而是 CTA 和用户阶段不匹配。

冷流量刚进来，你让他“立即购买”可能太重；B2B 用户刚了解方案，你让他“提交电话”也可能太重。相反，热流量已经准备购买，你还一直让他“了解更多”，也会浪费意图。

所以我会把 CTA 当成用户承诺强度来看：

| CTA | 承诺强度 | 适用场景 |
|---|---:|---|
| 立即购买 / 下单 | 高 | 热流量、价格清楚、信任充分 |
| 预约 Demo / 联系销售 | 高 | B2B、高客单价、复杂决策 |
| 免费试用 / 注册 | 中 | SaaS、工具、平台产品 |
| 下载资料 / 查看案例 | 中低 | 冷流量、教育型页面 |
| 查看价格 / 对比方案 | 中 | 有兴趣但还在评估 |
| 观看演示 / 了解流程 | 低 | 新产品、复杂产品、冷启动 |

## 4. 我沉淀的 12 个诊断维度

为了让诊断不靠感觉，我把落地页拆成 12 个维度。每个维度都可以打 0 到 5 分，并记录截图、数据或页面证据。

| 维度 | 我主要看什么 | 常见问题 |
|---|---|---|
| 来源意图匹配 | 页面有没有接住广告、搜索词、入口承诺 | 广告讲优惠，页面讲品牌故事 |
| 首屏价值主张 | 用户几秒内能不能理解价值 | 标题抽象、首屏无 CTA、利益不清楚 |
| 信息架构 | 内容顺序是否符合用户决策 | 证据太后、规则太散、重点被埋 |
| 可扫描性 | 小标题、列表、表格是否好读 | 大段文字、卖点堆叠、没有层级 |
| 可信证据 | 有没有让用户相信的材料 | 评价空泛、案例无结果、数据无口径 |
| 产品/服务解释 | 用户是否理解具体怎么用 | 只写功能名，不写场景和结果 |
| CTA 与行动路径 | 下一步是否明确、顺手、可追踪 | 按钮多、文案虚、点击后无反馈 |
| 风险解除 | 用户的顾虑是否被提前回答 | 价格、售后、隐私、退款不清楚 |
| 移动端体验 | 手机上是否好看、好点、加载快 | 首屏白屏、弹窗遮挡、按钮太小 |
| 合规与透明 | 是否存在夸大、隐藏、诱导 | 隐藏费用、默认勾选、虚假稀缺 |
| 数据追踪 | 能不能判断问题出在哪里 | UTM 丢失、事件重复、归因口径不明 |
| 品牌一致性 | 页面是否和品牌、产品、客服口径一致 | 广告、页面、客服说法不一样 |

这个框架最重要的不是“总分多少”，而是找出哪几个问题正在阻塞用户继续往下走。

## 5. 诊断维度一：来源意图匹配

这是我最优先看的维度，尤其在投放场景里。

用户点击之前已经被某个信息吸引。落地页要做的第一件事，就是让用户确认“我来对地方了”。

### 我会这样检查

| 检查项 | 判断方式 |
|---|---|
| 广告素材里的核心承诺是否出现在首屏 | 对照素材截图和页面首屏 |
| 搜索关键词是否被页面标题承接 | 对照 Search Term、Keyword、H1、副标题 |
| 优惠、价格、活动时间是否一致 | 对照广告、页面、购物车、Checkout |
| 商品图、款式、颜色是否一致 | 对照 Shopping、PMax、Meta 素材和商品页 |
| URL 参数是否保留 | 检查 UTM、ref、gclid、fbclid 是否丢失 |
| 用户阶段是否匹配 | 冷流量是否有解释，热流量是否有购买路径 |

### 我常见的判断

| 数据表现 | 我的判断方向 |
|---|---|
| CTR 高，CVR 低 | 素材吸引了点击，但页面没有承接素材角度 |
| CPC 正常，CPA 高 | 流量成本不一定是问题，先查页面和漏斗 |
| ATC 高，Purchase 低 | 商品兴趣存在，重点看结账前风险和费用 |
| 跳出高，停留短 | 首屏理解失败或加载体验差 |
| Google Ads 落地页体验低 | 关键词、广告、页面和导航需要重新对齐 |

## 6. 诊断维度二：首屏价值主张

首屏不需要把所有信息都说完，但必须让用户知道自己是否值得继续。

我不喜欢首屏只写很大的品牌口号，例如：

> 重新定义未来体验  
> 让增长更简单  
> 打造智能化解决方案

这些话不是完全不能用，但如果没有具体对象、场景和结果，用户很难判断和自己有什么关系。

### 我更倾向的表达方式

| 页面类型 | 更有效的表达结构 |
|---|---|
| 商品页 | 商品类别 + 使用场景 + 核心差异 |
| SaaS 官网 | 目标角色 + 关键任务 + 可感知结果 |
| B2B 服务页 | 行业/场景 + 解决方案 + 可信证据 |
| App 下载页 | 用户需求 + 核心功能 + 下载理由 |
| 平台服务页 | 用户任务 + 条件 + 开始入口 |

### 首屏检查清单

| 检查项 | 合格标准 |
|---|---|
| H1 是否具体 | 能说明产品/服务/任务，而不是只表达愿景 |
| 副标题是否补充价值 | 说明适用人群、场景、结果或差异 |
| 首屏是否有证据 | 评分、客户、案例、销量、认证、媒体等 |
| 主 CTA 是否清楚 | 用户知道点了以后会发生什么 |
| 移动端是否成立 | 手机首屏不被图片、弹窗、导航挤压 |

## 7. 诊断维度三：信息架构和可扫描性

我会把页面内容当成一次“说服过程”，不是内容堆叠。

多数页面可以按这个顺序组织：

1. 首屏讲清楚价值。
2. 说明用户问题或场景。
3. 展示产品/服务如何解决。
4. 给出证据。
5. 解释价格、版本、参数或流程。
6. 处理风险和异议。
7. 给出行动入口。
8. 用 FAQ 收尾。

这不是固定模板，但它符合大多数用户的决策顺序。

### 我会特别看小标题

如果只读页面小标题就能理解整页逻辑，这个页面通常不会太差。反过来，如果小标题都是：

- 核心优势；
- 为什么选择我们；
- 产品功能；
- 客户案例；
- 常见问题；

这种结构也能用，但还不够具体。更好的小标题应该直接回答用户问题，比如：

- 适合哪些使用场景？
- 和普通方案相比差在哪里？
- 从提交需求到上线需要多久？
- 价格包含哪些服务？
- 不适合哪些情况？

可扫描性不是把内容变短，而是让用户不用费力也能找到自己关心的信息。

## 8. 诊断维度四：可信证据

页面转化差，很多时候不是用户不需要，而是用户不信。

我会把证据分成几类：

| 证据类型 | 适合场景 | 好证据的标准 |
|---|---|---|
| 用户评价 | 电商、App、SaaS | 具体、有场景、有身份或来源 |
| 案例研究 | B2B、服务、SaaS | 有背景、问题、方案、结果 |
| 数据指标 | 工具、平台、广告服务 | 有口径、时间、样本或说明 |
| 产品截图/视频 | 工具、App、复杂服务 | 展示真实界面和流程 |
| 资质认证 | 金融、医疗、企业服务 | 可验证，且放在关键位置 |
| 价格/政策 | 电商、订阅、服务 | 提前披露，不藏在最后一步 |

我会避免只用一堆 Logo 或空泛好评来证明可信。比如“很多客户都说好”不如“某类客户在某个场景下解决了什么问题”。

### 可信证据应该靠近决策点

很多页面把案例、评价、售后、FAQ 全放在很下面，但用户在首屏或价格区就已经开始犹豫了。

所以我会建议把关键证据前置：

- 首屏附近放一句强证据；
- 价格附近放退款/保障；
- 表单附近放隐私说明；
- CTA 附近放低风险承诺；
- 商品详情附近放真实评价和使用场景。

## 9. 诊断维度五：产品和服务解释

很多页面会把功能写得很全，但用户还是不理解。原因通常是页面只写了“有什么”，没有写“怎么用、用完得到什么”。

### 我会用这张表检查

| 检查项 | 好的表现 | 差的表现 |
|---|---|---|
| 功能说明 | 功能和用户任务绑定 | 只堆功能名 |
| 使用场景 | 有角色、场景、前后对比 | 只有抽象口号 |
| 工作原理 | 用步骤、截图或流程图解释 | 用概念替代解释 |
| 输出结果 | 展示用户会得到什么 | 只说提升效率 |
| 限制条件 | 说明适合和不适合谁 | 承诺过满 |

如果是作品集展示，我会特别强调这一点：我不是只会改文案，而是会把产品能力翻译成用户能理解的任务语言。

## 10. 诊断维度六：CTA 和行动路径

CTA 的问题经常被简化成“按钮颜色”“按钮位置”。这些当然会影响点击，但更关键的是：这个动作对用户来说是不是合理。

### 我会看 7 件事

| 检查项 | 判断标准 |
|---|---|
| 主 CTA 是否明确 | 一个屏幕里不要有太多同级按钮 |
| CTA 文案是否具体 | “获取报价”比“提交”清楚 |
| CTA 是否匹配阶段 | 冷流量别一上来就强迫高承诺 |
| CTA 周围是否有风险解除 | 例如“无需信用卡”“1 个工作日内联系” |
| 表单字段是否必要 | 每个字段都要有业务理由 |
| 点击后是否有反馈 | 成功、失败、加载、下一步都要清楚 |
| 是否能追踪 | 点击、提交、购买、下载事件要能验证 |

### 表单页我会重点看

| 问题 | 影响 | 修改方向 |
|---|---|---|
| 字段过多 | 提交率下降 | 先收最低必要信息 |
| 字段名太内部 | 用户不知道怎么填 | 用用户语言改字段名 |
| 错误提示不清楚 | 用户无法修正 | 说明错在哪里、怎么改 |
| 没有隐私说明 | 用户不敢提交 | 说明用途、联系频率、隐私政策 |
| 提交后无确认 | 用户不知道是否成功 | 给确认页、邮件或下一步说明 |

## 11. 诊断维度七：风险解除

用户转化前一定会做风险评估，只是不同页面风险不同。

| 页面类型 | 用户最常见顾虑 | 页面应该回答 |
|---|---|---|
| 商品页 | 尺码、发货、退换、质量 | 尺码表、物流、退换、评价、保障 |
| 高客单价商品 | 值不值、会不会踩雷 | 材料、工艺、对比、案例、售后 |
| SaaS | 数据安全、迁移成本、取消 | 权限、认证、迁移流程、取消政策 |
| B2B 服务 | 费用、周期、效果、交付边界 | 报价方式、流程、案例、交付清单 |
| App | 是否占空间、是否好用、是否收费 | 核心功能、截图、评分、权限说明 |
| 订阅产品 | 是否自动续费、怎么取消 | 续费说明、提醒机制、取消路径 |

我的经验是：FAQ 不应该只是页面底部的“垃圾桶”。它应该承接用户最后几个不行动的理由。

## 12. 诊断维度八：移动端和加载体验

做投放时，我一定会用手机打开最终 URL，而不是只看电脑预览。

因为很多问题只有在真实移动端才会出现：

- 首屏图片加载太慢；
- 标题被导航挤掉；
- 弹窗遮住 CTA；
- 聊天插件挡住购买按钮；
- 视频自动播放导致页面卡顿；
- Instagram / Facebook WebView 打开样式异常；
- 跳转后 UTM 或 ref 丢失；
- Shopify 结账页和前页价格不一致。

### 我会优先检查

| 检查项 | 工具或方法 |
|---|---|
| 首屏是否 3 秒内可理解 | 真机打开并录屏 |
| CTA 是否可见可点 | 手机点击测试 |
| 页面是否稳定 | PageSpeed Insights / Lighthouse |
| LCP / INP / CLS 是否异常 | Google Search Console / PageSpeed Insights |
| 参数是否保留 | 手动点击带参 URL |
| 弹窗是否影响转化 | 真机检查关闭成本 |

性能不是技术团队自己的事。只要它影响用户理解和行动，它就是落地页内容诊断的一部分。

---

## 13. 诊断维度九：合规和透明

这个维度我会放得很靠前，尤其是广告投放、订阅、电商、金融、医疗、减肥、教育、招聘等场景。

高风险问题包括：

- 夸大收益；
- 虚假稀缺；
- 隐藏费用；
- 默认勾选；
- 取消路径复杂；
- 自动续费不清楚；
- 隐私和数据用途不透明；
- 广告、页面、Checkout 价格不一致；
- 敏感行业使用绝对化承诺。

这类问题短期可能带来点击或转化，但长期会带来审核、投诉、退款、拒付和品牌风险。

我会把合规问题分成两类：

| 类型 | 处理方式 |
|---|---|
| P0 风险 | 页面上线或投放前必须处理，例如价格不一致、无法取消、虚假承诺 |
| P1 风险 | 本轮优化处理，例如条款不够清楚、隐私说明位置太靠后 |

---

## 14. 诊断维度十：数据追踪和归因

如果数据不准，页面诊断很容易变成猜。

我会先确认这些数据能不能看：

| 数据 | 用途 |
|---|---|
| 广告花费、点击、CTR、CPC、CPM | 判断入口质量和流量成本 |
| CVR、CPA、ROAS、转化价值 | 判断商业结果 |
| Landing Page、Source/Medium、Campaign | 判断页面和渠道表现 |
| Scroll、CTA Click、Form Submit、ATC、Purchase | 判断漏斗阻塞点 |
| Search Term / Keyword | 判断搜索意图 |
| 热图、录屏 | 判断用户真实行为 |
| 客服问题、评论、退款原因 | 判断页面没有回答什么 |

### 我会特别提醒口径问题

投放分析里经常出现“平台说有效，GA4 看起来没效果”或“Shopify 订单和广告后台对不上”。这不一定是谁错了，而是口径不同。

| 差异来源 | 可能影响 |
|---|---|
| 归因模型不同 | Meta、Google Ads、GA4 归因结果不同 |
| 转化窗口不同 | 1 天点击、7 天点击、浏览型转化都会影响 |
| 事件重复 | Purchase 可能被触发多次 |
| 跨域问题 | 支付、预约、第三方表单导致来源丢失 |
| UTM 丢失 | 流量被记成 direct 或 referral |
| 退款和币种 | 后台收入与广告转化价值不一致 |

所以在作品集里，我会主动写清楚“我用什么口径判断”，这会比只展示一个漂亮增长数字更可信。

---

## 15. 诊断维度十一：品牌一致性

短期转化页常常只关注点击和成交，但一个页面如果和品牌、客服、App、产品实际体验不一致，后续会产生很多隐性成本。

我会检查：

- 广告承诺和页面承诺是否一致；
- 页面价格和结账价格是否一致；
- 页面政策和客服话术是否一致；
- 产品截图和真实界面是否一致；
- 页面语气是否符合品牌；
- 是否为了转化牺牲了长期信任。

这个维度适合放在作品集里，因为它能体现你不是只看短期 ROAS，也能理解品牌、产品和用户体验之间的关系。

## 16. 我的实际诊断流程

### 16.1 先还原用户进来前的预期

我不会一上来就改页面，而是先回到用户点击之前，看他是被什么信息带进来的。因为很多落地页问题不是页面里某个模块写得不好，而是入口承诺和页面承接没有对上。

| 我会看什么 | 为什么看 |
|---|---|
| 广告素材、关键词、邮件标题、站内入口 | 判断用户点进来时期待看到什么 |
| 用户所在渠道和设备 | 判断他是主动搜索、被素材吸引，还是已有一定认知 |
| 最终 URL 和跳转链路 | 确认 UTM、ref、gclid、fbclid 等参数没有丢 |
| 移动端首屏 | 还原大多数广告流量真正看到的第一屏 |

这一步的输出不是一堆建议，而是一句话判断：

> 这个页面有没有接住用户原本点进来的理由？

### 16.2 再拆页面有没有完成“理解、相信、行动”

确认入口预期后，我会沿着用户阅读页面的顺序往下看。这里不只是看视觉，而是看页面有没有让用户完成三个动作：理解、相信、行动。

| 诊断层 | 我会看什么 |
|---|---|
| 理解 | 首屏是否讲清楚这是什么、给谁用、解决什么问题 |
| 相信 | 评价、案例、数据、价格、政策、资质是否足够可信 |
| 行动 | CTA 是否顺手，表单/购买/注册路径是否清楚 |
| 风险 | 运费、退款、隐私、订阅、隐藏费用、夸大承诺是否提前说明 |
| 数据 | GA4、广告平台、Pixel、热图、录屏能否支持判断 |

我会把问题尽量写成可验证的判断，而不是主观评价。比如不写“页面说服力不够”，而是写：

> 广告素材强调限时优惠，但首屏没有出现优惠信息，用户需要滚动到第三屏才看到活动规则。这会增加确认成本，也可能拉低移动端 CVR。

### 16.3 最后把问题排成执行顺序

诊断最后一定要落到行动优先级。页面通常不会只有一个问题，如果所有建议都同时做，就很难判断到底是哪一次修改起了作用。

我会按这几个标准排序：

| 判断标准 | 说明 |
|---|---|
| 影响范围 | 是否影响主要流量、主要设备、主要转化路径 |
| 修复确定性 | 是明显错误，还是需要实验验证 |
| 业务风险 | 是否涉及审核、价格、隐私、退款、订阅等风险 |
| 执行成本 | 文案、结构、开发、埋点、设计分别需要多少资源 |
| 验证方式 | 能否用 CVR、CPA、ROAS、表单率、加购率、录屏等验证 |

最后的输出应该是一组可以推进的动作，而不是一句“建议优化落地页”。

## 17. 我会如何给优化建议分优先级

我不会把所有建议都列成“建议优化”。这样对执行没有帮助。

我会分成 5 类：

| 类型 | 适用情况 | 示例 |
|---|---|---|
| 立即执行 | 明显影响转化、审核、追踪或用户完成任务 | 页面打不开、CTA 不可点、价格不一致、UTM 丢失 |
| 观察等待 | 数据还不够，或波动可能来自学习期/样本不足 | 新广告组刚上线 1 天，样本过少 |
| 小预算测试 | 有明确假设，但需要验证 | 改首屏标题、换证据顺序、减少表单字段 |
| 暂停或降预算 | 页面问题会浪费预算或带来风险 | 支付异常、移动端白屏、活动价不一致 |
| 需要补充数据 | 当前无法判断 | 缺 GA4、缺广告素材、缺转化定义、缺利润数据 |

如果是投放落地场景，我会优先用这个排序：

> 优先级 = 花费占比 × 转化损失 × 修复确定性 × 风险等级

这样可以避免把时间花在低流量、低影响页面上。

## 18. 广告投放场景的专项检查

### 18.1 Meta Ads

| 检查项 | 我会看什么 |
|---|---|
| 素材承诺是否被首屏承接 | 折扣、痛点、场景、达人内容、产品款式 |
| Post ID / 达人贴文是否一致 | 用户点击后是否能找到同款、同优惠、同说法 |
| UTM 和 ref 是否正确 | 没有多个 `?`、多个 `ref=`、中文符号、空格 |
| WebView 是否正常 | Instagram / Facebook 内打开是否加载、是否遮挡 |
| Pixel / CAPI 是否触发 | ViewContent、ATC、Purchase 是否去重 |
| 素材衰减后页面是否还匹配 | 新素材角度变了，页面也要同步承接 |

### 18.2 Google Ads

| 检查项 | 我会看什么 |
|---|---|
| Search 关键词和页面标题是否匹配 | 广告组越细，页面越要精准 |
| Landing Page Experience 是否异常 | 结合关键词、广告、页面内容、导航、速度检查 |
| PMax / Shopping 信息是否一致 | 标题、价格、图片、库存、促销 |
| 页面是否易导航 | 用户能否找到相关内容和下一步 |
| Final URL 和 Display URL 是否一致 | 避免 destination mismatch |
| gclid 是否保留 | 重定向、跨域、支付都要检查 |
| 转化动作是否重复 | Purchase、Lead、Signup 不要混乱 |

### 18.3 GA4

| 检查项 | 我会看什么 |
|---|---|
| Landing page + query string 是否可用 | 用于区分 UTM 和 ref |
| Key Event 是否定义正确 | 不把低价值事件当核心转化 |
| Source / Medium 是否异常 | direct、referral、paid social、paid search 是否合理 |
| 跨域是否配置 | Shopify、支付、预约工具、第三方表单 |
| 事件参数是否完整 | value、currency、item_id、campaign、content |
| 归因窗口是否说明 | 分析时必须标注口径 |

## 19. 可以直接放进作品集的案例结构

如果我把这套方法用在一个真实项目里，我会这样展示：

### 19.1 项目背景

写清楚页面类型、业务目标、流量来源和核心 KPI。

示例：

> 这是一个承接 Meta 和 Google Ads 流量的商品落地页，主要目标是提升 Purchase CVR 和 ROAS。页面在素材点击率正常的情况下，购买转化偏低，且移动端跳出率高。

### 19.2 问题定义

不要写“页面不好看”，而是写成可验证的问题。

示例：

> 诊断后发现，页面的主要问题不是素材点击不足，而是广告承诺和首屏内容不一致。素材强调使用场景和限时优惠，但首屏主要展示品牌图和泛化口号，用户需要继续滚动才能看到价格、优惠和保障信息。

### 19.3 诊断方法

展示你的方法论，而不是只展示结果。

可以放：

- 12 维度评分表；
- 首屏截图标注；
- 漏斗数据；
- 热图或滚动数据；
- 广告素材和页面对照；
- 问题优先级矩阵。

### 19.4 核心发现

建议只列 3 到 5 个，不要把所有问题都堆出来。

示例：

| 发现 | 证据 | 影响 |
|---|---|---|
| 首屏没有承接广告承诺 | 素材强调折扣，首屏没有出现折扣信息 | 用户确认成本高 |
| 信任证据位置太靠后 | 评价和退换政策在页面底部 | 冷流量不敢购买 |
| CTA 文案过于泛 | 按钮写“了解更多” | 购买意图没有被及时承接 |
| 移动端弹窗遮挡按钮 | 真机录屏显示首次进入弹窗挡住首屏 | 增加跳出 |

### 19.5 优化方案

按页面模块写，而不是按个人想法写。

| 模块 | 优化方向 |
|---|---|
| 首屏 | 重写 H1、副标题、利益点和主 CTA |
| 证据区 | 前置评价、案例、评分、保障 |
| 商品/服务解释 | 用场景和结果替代功能堆叠 |
| 风险解除 | 增加运费、退换、隐私、取消等说明 |
| 表单/购买路径 | 减少字段、优化错误提示、检查支付 |
| 数据追踪 | 修复 UTM、Pixel、GA4 事件 |

### 19.6 结果复盘

有数据就展示数据，没有数据也要说明验证方式和限制。

可以看：

- CVR；
- CPA；
- ROAS；
- 转化价值；
- 表单提交率；
- ATC Rate；
- Checkout Rate；
- 页面滚动深度；
- CTA 点击率；
- 跳出率；
- 客服咨询问题变化。

如果样本不足，可以诚实写：

> 当前样本不足以证明最终转化提升，因此本轮只判断中间行为指标，例如首屏停留、CTA 点击和表单开始率。购买结果需要继续观察。

这比硬写一个不可靠的增长结果更专业。

## 20. 可复制的诊断结论模板

### 20.1 投放页版本

> 本轮诊断显示，页面当前主要问题不是流量不足，而是来源意图和首屏承接不足。广告素材强调具体场景和优惠，但落地页首屏更偏品牌表达，用户需要继续滚动才能找到价格、保障和购买理由。建议优先调整首屏标题、副标题、核心证据和 CTA，并将物流、退换、优惠规则提前到首屏下方。验证上，建议观察移动端 CVR、ATC Rate、Checkout Rate、CPA 和 ROAS，至少保留 7 天或达到足够样本后再判断是否放量。

### 20.2 SaaS / 工具页版本

> 页面目前能展示产品能力，但对新用户的任务承接还不够直接。用户能感受到产品功能很多，但难以快速判断自己应该从哪个场景开始、能得到什么结果、是否需要付费或预约。建议将页面从功能罗列改为任务引导：首屏明确目标用户和核心任务，中段按场景拆分能力，证据区展示真实界面和客户案例，底部提供试用、文档、价格和联系路径。

### 20.3 B2B 线索页版本

> 当前页面的线索转化阻力主要来自信任证据不足和表单承诺过高。页面要求用户提交较多信息，但在提交前没有充分解释交付范围、案例结果、响应时间和隐私用途。建议先降低首个行动门槛，例如提供案例下载、报价范围或预约说明，同时在表单附近增加隐私说明和提交后流程。

### 20.4 服务办理页版本

> 页面当前的问题不是说服力不足，而是任务路径不够清楚。用户进入页面后需要判断自己是否符合条件、需要准备什么材料、从哪里开始、提交后多久有结果。建议按用户办理顺序重排内容：适用条件、准备材料、办理步骤、常见错误、开始按钮、结果查询和客服入口。

## 21. 我希望作品集传达的能力

这套经验沉淀最终想表达的不是“我会做落地页优化”，而是更具体的能力：

1. 我能把页面问题放回完整转化链路里看，而不是只看视觉。
2. 我能从广告素材、关键词、用户意图和页面内容之间找断点。
3. 我能把抽象的“页面不转化”拆成可检查、可排序、可验证的问题。
4. 我能同时考虑短期投放结果和长期用户信任。
5. 我能用数据、截图、用户行为和业务目标支撑建议。
6. 我知道什么时候该立即改，什么时候该测试，什么时候样本不足不能下结论。

对作品集来说，这比单纯展示一个漂亮改版更有价值。因为成熟团队更关心的是：你能不能定义问题、组织证据、推动决策，并把优化结果和业务指标连起来。
